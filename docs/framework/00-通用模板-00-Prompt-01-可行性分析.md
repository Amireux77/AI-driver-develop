# 约束与要求

**要求** :
- 结构完整 : 上表 **全部章节** 须出现
- 内容要求 : 如测试主题有变体,则需要在对应点指出变体
- 章节要求 : 每个小节标题后面都需要加分割线,章标题前后均不加分割线
  - 文首元信息 : 以引用形式给出,包含 文档版本、编写日期、编写人员、文档定位、输入来源、关联文档、文档说明,引用内不用表格
  - 文档约定 : 以表格形式给出,表格列为 状态标记(以统一编号体系为准,只列举本文档中出现的符号,居中对齐)、含义(状态标记的含义)
  - 第一章 文档目的与范围
    - 1.1 目的
    - 1.2 范围
  - 第二章 背景与问题陈述
    - 2.1 业务背景 : 介绍对应项目的概念、立项的背景、Xugu 的期待
    - 2.2 不立项的风险
  - 第三章 方法与假设
    - 3.1 对标方法
    - 3.2 关键假设 : 以表格形式给出,表格列为 假设编号(`Ax`,加粗居中对齐)、假设内容、当前状态
    - 3.3 信息来源
  - 第四章 竞品能力并集 : 以表格形式给出,表格列为 分析项(该项目在各数据库中的功能并集)、Oracle(居中对齐)、PostgreSQL(居中对齐)、MySQL(居中对齐)、功能说明 ; 三个数据库列内容为文档约定中的状态标记
  - 第五章 虚谷现状与能力映射
    - 5.1 并集项 → 虚谷映射表 : 以表格形式给出,表格列为 并集项(取第四章的并集项)、虚谷(说明是支持、不支持、部分支持)、状态(输入文档约定中的对应符号,居中对齐)
    - 5.2 虚谷差异化优势
  - 第六章 虚谷缺失或不等价项 : 以表格形式给出,表格列为 编号(`Gxx`,加粗居中对齐)、能力、主要来源库、虚谷现状、迁移/测试影响
  - 第七章 缺口可行性分析 : 每小节标题为缺失项编号(`Gxx`),无文字赘述;每小节内容以表格形式给出,表格列为 四库实测(四个数据库实际运行的测试结果)、可行性(低/中/高)、测试要点、立项建议
  - 第八章 分集建议与测试域 : 以表格形式给出,表格列为 分集编号(`Lx`,加粗居中对齐)、需求分为、对应测试域(`TD-xx`,加粗居中对齐)
  - 第九章 风险、依赖与约束
    - 9.1 风险 : 以表格形式给出,表格列为 风险项、影响范围、缓解方式
    - 9.2 环境依赖
- 逻辑内容 : 章节自洽
- 内容详尽 : 竞品并集与虚谷映射表须直观 ; 只对比 Oracle、PG、MySQL
- 测试结论 : 明确是否可立项、建议分集方式、测试域表含优先级建议
- **工作流程** :
  1. **定位仓库** : 在含 `run.ps1`、`config/`、`db-compat-tester/` 的**本项目根目录**工作 ; Agent 模式须**实际执行**终端,不得只列命令

  5. **本阶段产物** : 仅 Markdown 可行性文档 + 可选连通/矩阵报告 ; **不** 在 `scripts/` 批量生成 SQL (SQL 在用例设计阶段产出)
