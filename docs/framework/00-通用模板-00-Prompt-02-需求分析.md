# 约束与要求

**要求** :
- 结构完整 : 上表**通用章节**必选 ; DML 或数据类型正文块二选一展开 , 不得混用两套骨架
- 逻辑内容 : 需求清单与验收索引集中 ; 正文详述不重复罗列 ; 引用用《文档名》或「本小节」
- 编号列表 : **FR-xxx**、**GR-xxx**、**AC-xx**、**TD-xx**、**A** / **Q** / **L** ; 数据类型可加 **T01～Tn** 变体
- 文档信息 : 文首引用含适用范围、不含范围 ; 文档说明中约定 **本文** / **可行性** / **设计文档** 等前缀 ( 无设计文档则删相关表述 )
- 内容详尽 : 关键对照表含 Oracle、PostgreSQL、MySQL、Xugu 四列 ( 数据类型章 ) ; DML 章以 Xugu 行为 + 三库迁移说明为主
- 包含内容 : 文首状态符号 ✔️ / ❌ / ➖ / ⚠️ / 🔺
- 库对比 : 结合技能集与《{可行性文件}》 , 列出三库已有而 Xugu 未实现或不等价项并写成 **GR-xxx**
- 验收 : **AC-xx** 与 **FR** / **GR** 、**TD-xx** 可双向追溯
- 版本升级 : 仅当存在《版本比对》文档时 , 在模板中保留「版本升级与迁移」相关章节 , 否则删除该整块
- **工作流程** :
  1. **定位仓库** : 在含 `run.ps1`、`config/`、`db-compat-tester/` 的**本项目根目录**工作 ; Agent 模式须**实际执行**终端 , 不得只列命令
  2. **环境预检** : 检查 `config/db.properties`、`lib/jdbc/*.jar`、`db-compat-tester/target/db-compat-tester-1.0.0-jar-with-dependencies.jar` ; 缺项则说明 , 可仍写文档 , 不得伪造 `reports/` 内容
  3. **文档落盘** : 按本章「输出格式」写入 `docs/<主题>/` ( 如 `docs/insert/INSERT-02-需求分析.md` ) , 替换 `{输出文件}`、`{系列}` 占位 ; 须已读同主题《{可行性文件}》
  4. **调用项目验证** ( 按需 ) : `.\run-connect.ps1` 或 `.\run.ps1 matrix/<矩阵>.json` 核对需求描述 , 报告在 `reports/` ; 有创表则验证后删除
  5. **本阶段产物** : 仅需求分析 Markdown ; **不**在 `scripts/` 生成用例 SQL

**约束** :
- 文章中的 `,` , `:` , `/` 等均使用英文并且前后各有一个空格 ; `" "` , `()` , `<>` , `** **` 以及行内代码块等在前后各加一个空格,如果配合使用则两个符号之间不用空格隔开,例如 : `**()**` 表示包括括号在内的内容加粗,文章中不要出现任何的中文标点符号
- 对于编号列表,要在文档开始以表格的形式总览,在后续章节中详细说明
- 表格中的编号列居中对齐,文档约定的符号例如 ✔️ 等也居中对齐
- 当有编号的时候,在文档最前面的章节要有总览,包含编号、说明,之后在后续章节中详细展开描述
- 编号命名须符合《00-统一编号体系》, 规则如下 :
  - **主题不写进编号** : 测试主题已由输出文件名表达 , 追溯编号**禁止**再嵌入主题段
  - **允许的前缀** : `FR` `GR` `PD` `AC` `TD` `GS` `T` 等
  - **格式示例** : `FR-001` , `GR-015` , `AC-03`
  - **跨文档引用** : 须带文档名
- 只对比 Oracle、PG、MySQL ; 不做缺口是否上线的最终决策
- 通过运行当前项目中的 Java 程序连接四个库来验证描述需求,如果有创表语句,验证结束之后删除表,验证报告存放在 reports 目录中
