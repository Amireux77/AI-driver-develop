# 约束与要求

**要求** :
- 结构完整 : 依据上述必备章节编写文档
- 逻辑内容 : 需求清单与验收索引集中 ; 正文详述不重复罗列
- 内容要求 : 如测试主题有变体,则需要在对应需求点指出变体
- 章节要求 : 每个小节标题后面都需要加分割线,章标题前后均不加分割线
  - 文首元信息 : 以引用形式给出,包含 文档版本、编写日期、编写人员、文档定位、输入来源、关联文档,引用内不用表格
  - 第一章 需求范围与追溯 : 
    - 1.1 需求目的
    - 1.2 范围边界
    - 1.3 测试分层 : 以表格形式给出,表格列为 分层号(`Lx`,加粗居中对齐)、分层标准、需求范围、测试域(对应的测试域代码 `TD-xx`)
    - 1.4 虚谷差异化约束
  - 第二章 需求清单 : 每小节内容以表格形式给出,
    - 2.1 Xugu 已实现需求功能需求 , 要与 `{可行性文件}` 中竞品能力并集中 Xugu 已实现能力相同;表格列为 需求编号 (功能需求 `FR-xxx`,缺口需求 `GR-xxx`,加粗居中对齐)、需求说明(一句话简短说明)、优先级(`Px`,加粗居中对齐)、对应风险(预计会产生的风险)
    - 2.2 Xugu 未实现需求和缺口需求; 要与 `{可行性文件}` 中竞品能力并集中 Xugu 未实现能力相同 ; 表格列为 需求编号 (功能需求 `FR-xxx`,缺口需求 `GR-xxx`,加粗居中对齐)、需求说明(一句话简短说明)、优先级(`Px`,加粗居中对齐)、对应风险(预计会产生的风险)
  - 第三章 未实现需求详细 : 每小节内容以表格形式给出,每小节标题为未实现需求编号无后续文字赘述,表格列为 需求竞品来源 (功能来源数据库) 、需求内容 (详细说明) 、示例语句 、预期结果
  - 第四章 验收标准与通过准则 : 
    - 4.1 验收项目 : 以表格形式给出,按优先级(`Px`)排序,表格列为 验收编号(`AC-xx`,加粗居中对齐)、验收项(具体验收内容)、测试域(验收项目对应的测试域编号 `TD-xx`,加粗居中对齐)、验收需求编号(`FR-xxx`或`GR-xxx`,加粗居中对齐)
    - 4.2 通过准则 : 包含 各优先级的通过准则(如通过率、不阻塞等等)、缺口未立项的通过准则
- **工作流程** :
  1. **定位仓库** : 在含 `run.ps1`、`config/`、`db-compat-tester/` 的 **本项目根目录** 工作 ; Agent 模式须 **实际执行** 终端,不得只列命令
  2. **环境预检** : 检查 `config/db.properties`、`lib/jdbc/*.jar`、`db-compat-tester/target/db-compat-tester-1.0.0-jar-with-dependencies.jar` ; 缺项则说明,可仍写文档,不得伪造 `reports/` 内容
  3. **文档落盘** : 按本章「输出格式」写入 `docs/<主题>/` (如 `docs/insert/INSERT-02-需求分析.md`),替换 `{输出文件}`、`{系列}` 占位 ; 须已读同主题《{可行性文件}》
  4. **调用项目验证** (按需) : `.\run-connect.ps1` 或 `.\run.ps1 matrix/<矩阵>.json` 核对需求描述,报告在 `reports/` ; 有创表则验证后删除
  5. **本阶段产物** : 仅需求分析 Markdown ; **不** 在 `scripts/` 生成用例 SQL

**约束** :
- 文章中的 `,`,`:`,`/` 等均使用英文并且前后各有一个空格 ; `" "`,`()`,`<>`,`** **`整体而非单个 `"` 或 `(` 以及行内代码块等在前后各加一个空格,如果配合使用则两个符号之间不用空格隔开,例如 : `**()**` 表示包括括号在内的内容加粗,文章中不要出现任何的中文标点符号,加粗文字与普通文字之间要有空格,例如`小明 **张三** 大明`
- 表格中的编号列居中对齐,文档约定的符号例如 ✔️ 等也居中对齐,所有编号都要加粗
- 编号命名须符合《00-统一编号体系》, 规则如下 :
  - **主题不写进编号** : 测试主题已由输出文件名表达,追溯编号 **禁止** 再嵌入主题段
  - **允许的前缀** : `FR` `GR` `PD` `AC` `TD` `GS` `T` 等
  - **格式示例** : `FR-001`,`GR-015`,`AC-03`
  - **跨文档引用** : 文档之间无耦合性,不互相引用,如需表明其余文档,只需要说明文档名即可,不出现引用章节名
- 只对比 Oracle、PG、MySQL ; 不做缺口是否上线的最终决策
- 通过运行当前项目中的 Java 程序连接四个库来验证描述需求,如果有创表语句,验证结束之后删除表,验证报告存放在 reports 目录中
- 只按要求来进行,不要出现要求之外的内容
