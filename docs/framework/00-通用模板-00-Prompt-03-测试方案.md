# 约束与要求

**要求** :
- 结构完整,必须包含已实现和未实现需求的测试方案
- 内容要求 : 如测试主题有变体,则需要在对应测试点指出变体
- 章节要求 : 每个小节标题后面都需要加分割线,章标题前后均不加分割线
  - 文首元信息 : 以引用形式给出,包含 文档版本、编写日期、编写人员、文档定位、关联文档、**主测库**、**对标库** (无则写「无),引用内不用表格
  - 第一章 测试域总览 : 每小节以表格形式给出 , 章首先列出所有测试域,以表格形式给出,表格列为 测试域编号(`TD-xx`,加粗居中对齐)、测试内容、优先级(`Px`,加粗居中对齐)、需求依据(摘要)
    - 1.1 **已实现需求** 与 **测试域** 覆盖 : 表格列为 需求编号 (`FR-xxx`,与需求分析的编号对齐,加粗居中对齐)、需求摘要(介绍需求内容)、对应测试域(`TD-xx`,加粗居中对齐)、测试优先级(`Px`,加粗居中对齐)、设计依据 (`D-xx`,与设计方案中干的设计编号对齐,加粗居中对齐)
    - 1.2 **未已实现需求** 与 **测试域** 覆盖 : 表格列为 需求编号 (`FR-xxx`,`GR-xxx`,`AC-xx`,`Gxx`,与需求分析的编号对齐,加粗居中对齐)、需求摘要(介绍需求内容)、对应测试域(`TD-xx`,加粗居中对齐)、测试优先级(`Px`,加粗居中对齐)、设计依据 (`D-xx`,与设计方案中干的设计编号对齐,加粗居中对齐)
    - 1.3 **测试分层** : 表格列为 分层号 (`Lx`,加粗居中对齐)、分层范围、对应测试域
  - 第二章 目标范围及通过准则 : 
    - 2.1 测试目标与范围
    - 2.2 通过准则,以表格形式给出,表格列为 优先级别(`Px`,加粗居中对齐)、含义、通过要求(对应测试域的通过要求)
  - 第三章 测试点编号规则 : 
    - 3.1 用例编号格式 : 规范格式 **`!4 XG-TYPE-<主题>-<域代码>-<序号>`** + 示例
    - 3.2 域代码与测试域编号对照 : 以表格形式给出,表格列为 域代码 (`VAL`、`SEL`、...,加粗居中对齐)、对应的测试域编号(`TD-xx`,加粗居中对齐)、含义(解释域代码的含义)
    - 3.3 用例设计编写表头 : 以表格形式给出,表格列为 列名(编号、优先级、变体、维度、测试点、需求依据、设计依据、验收要点)、填写说明(说明这些内容如何填写,填写的内容示例)
  - 第四章 各测试域说明 : 每小节标题为测试域编号 `TD-xx`,无文字赘述; 每小节内容以表格形式给出,表格列为 需求依据(需求分析文档对应需求编号 `FR-xxx` 或 `GR-xxx`,加粗)、设计依据(`D-xx`,设计文档对应设计编号,无填 `-`,加粗居中对齐)、域级验收(域级编号及通过准则)、用例设计方向
  - 第五章 测试环境 : 以表格形式给出,表格列为 项(主测库、对标库、客户端/驱动、Schema/数据库、数据准备、字符集/时区、事务隔离)、要求(项的具体内容)
  - 第六章 风险与假设 : 以表格的形式给出,表格列为 编号(`Axx`,加粗居中对齐)、风险、影响范围(影响的测试域编号`TD-xx`)、缓解(降低风险的推荐方法)
  - 第七章 交付物 : 以表格形式给出,表格列为 交付物、说明
- 范围 : 只写方案 , 不编造需求未声明的测试域
- **工作流程** :
  1. **定位仓库** : 在含 `run.ps1`、`config/`、`db-compat-tester/` 的**本项目根目录**工作 ; Agent 模式须**实际执行**终端 , 不得只列命令
  2. **环境预检** : 检查 `config/db.properties`、`lib/jdbc/*.jar`、`db-compat-tester/target/db-compat-tester-1.0.0-jar-with-dependencies.jar` ; 缺项则说明 , 可仍写文档 , 不得伪造 `reports/` 内容
  3. **阅读框架** : 必读 `docs/framework/00-测试维度框架.md` 及同主题《{系列}-02-需求分析》
  4. **文档落盘** : 按本章「输出格式」写入 `docs/<主题>/` (如 `docs/insert/INSERT-03-测试方案.md`) , 替换 `{输出文件}`、`{系列}` 占位
  5. **调用项目验证** (按需) : `.\run-connect.ps1` 或 `.\run.ps1 matrix/<矩阵>.json` , 报告在 `reports/` ; 有创表则验证后删除
  6. **本阶段产物** : 仅测试方案 Markdown ; **不**在 `scripts/` 生成用例 SQL

**约束** :
- 文章中的 `,` , `:` , `/` 等均使用英文并且前后各有一个空格 ; `" "` , `()` , `<>` , `** **` 以及行内代码块等在前后各加一个空格,如果配合使用则两个符号之间不用空格隔开,例如 : `**()**` 表示包括括号在内的内容加粗,文章中不要出现任何的中文标点符号
- 表格中的编号列居中对齐,文档约定的符号例如 ✔️ 等也居中对齐,所有编号都要加粗
- 编号命名须符合《00-统一编号体系》, 规则如下 :
  - **主题不写进 TD** : 测试域 **TD** 不含 INSERT、INTERVAL 等主题段
  - **用例编号** : `XG-TYPE-<主题缩写>-<域代码>-<序号>` , 示例 `TD-08` , `XG-TYPE-INSERT-VAL-001`
  - **跨文档引用** : 文档之间无耦合性,不互相引用,如需表明其余文档,只需要说明文档名即可,不出现引用章节名
- 主测库为 Pass/Fail 判定依据 ; 对标库差异仅在方案中声明的跨库 TD 登记
- 通过运行当前项目中的 Java 程序连接四个库来设计测试方案,如果有创表语句,验证结束之后删除表,验证报告存放在 reports 目录中
- 只按要求来进行,不要出现要求之外的内容
