# 约束与要求

**要求** :
- 结构完整 : 上表**全部章节**须出现
- 逻辑内容 : 不重复需求/设计原文 , 只写测试点与验收要点 ; 引用《需求分析》《测试方案》等文档名
- 编号列表 : `XG-TYPE-<主题缩写>-<域代码>-<序号>` ; 与 FitNesse `!4` 行一致 ; 测试域写在用例标题或小节标题中
- 文档信息 : 含主测库、自动化框架与脚本根目录 ( 若有 )
- 内容详尽 : 各测试域用例表使用**统一表头** ( 见 §3.3 )
- 包含内容 : 与测试方案 **TD** 一一对应 ; **P0** 覆盖各变体边界与至少一条主路径
- 矩阵用例 : 一条编号覆盖多组合时 , 在「说明」列列出维度
- 金样 : 若有回归金样 **GS-xxx** , 单独列出并与方案回归范围一致
- 精简 : 表中不写长 SQL , 写 `!4` 行摘要或文件路径
- **工作流程** :
  1. **定位仓库** : 在含 `run.ps1`、`tools/new-sql-script.ps1`、`config/` 的**本项目根目录**工作 ; Agent 模式须**实际执行**终端 , 不得只列命令
  2. **环境预检** : 检查 `config/db.properties`、`lib/jdbc/*.jar`、`db-compat-tester/target/db-compat-tester-1.0.0-jar-with-dependencies.jar` ; 缺项则说明 , 可仍写文档 , 未齐前勿伪造报告
  3. **文档落盘** : 按本章「输出格式」写入 `docs/<主题>/` ( 如 `docs/insert/INSERT-04-用例设计.md` ) , 须已读同主题《{系列}-02》《{系列}-03》
  4. **SQL 落盘** : 见《00-通用模板-00-Prompt-SQL生成》— **§16.1 已实现** → `scripts/<主题>_TYPE/implemented/` ( 语法以 [docs.xugudb.com](https://docs.xugudb.com/) 为准 ) ; **§16.2 / GR** → `…/gap/` ; 脚手架 `-Scope implemented|gap` ( **INTERVAL** 可用 `-VariantCount 13` ; **INSERT/UPDATE** **禁止** `-VariantCount` )
  5. **用例表追溯** : 「说明」列写全路径 , 如 `scripts/INSERT_TYPE/implemented/01-Val.sql` 内 `!4 …` ; 每条 `CREATE TABLE` 须有 `DROP`
  6. **四库矩阵** ( 按需 ) : 在 `matrix/` 编写/更新 `*.json` , 执行 `.\run.ps1 <矩阵名>.json` , 报告在 `reports/`
  7. **连通预检** : 执行 `.\run-connect.ps1` 确认四库可达后再跑矩阵

**约束** :
- 文章中的 `,` , `:` , `/` 等均使用英文并且前后各有一个空格 ; `" "` , `()` , `<>` , `** **` 以及行内代码块等在前后各加一个空格,如果配合使用则两个符号之间不用空格隔开,例如 : `**()**` 表示包括括号在内的内容加粗,文章中不要出现任何的中文标点符号
- 对于编号列表,要在文档开始以表格的形式总览,在后续章节中详细说明
- 表格中的编号列居中对齐,文档约定的符号例如 ✔️ 等也居中对齐
- 当有编号的时候,在文档最前面的章节要有总览,包含编号、说明,之后在后续章节中详细展开描述
- 编号命名须符合《00-统一编号体系》, 规则如下 :
  - **GS** 不含主题段 ; 用例编号含 `TYPE-<主题>` 段 , 域代码与 TD 对应
  - **格式示例** : `XG-TYPE-INSERT-VAL-001` , `GS-001`
  - **废弃编号不重用**
- 不要臆测需求未声明的语法或函数
- 通过运行当前项目中的 Java 程序连接四个库来设计用例,如果有创表语句,验证结束之后删除表,验证报告存放在 reports 目录中
