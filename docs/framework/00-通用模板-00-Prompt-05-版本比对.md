# 约束与要求

**要求** :
- 结构完整 : 上表 **必备章节** 须全部出现
- 章节要求 : 每个小节标题后面都需要加分割线,章标题前后均不加分割线
  - 文首元信息 : 以引用形式给出,包含 版本比对(新版VS旧版)、旧版、新版、编写日期、编写人员、比对范围
  - 文档约定 : 以表格形式给出,表格列为 状态标记(以统一编号体系为准,只列举本文档中出现的符号,居中对齐)、含义(状态标记的含义)
  - 第一章 总览比对 : 以表格形式给出,表格列为 变更编号(`Cxx`,加粗居中对齐)、对比维度、是否变更(使用文档约定中的符号,居中对齐)、风险
  - 第二章 变更动机 : 以表格形式给出,表格列为 变更编号(`Cxx`,加粗居中对齐)、变更动机(详述变更的动机是什么)
  - 第三章 细节比对 : 每小节标题为变更编号 `Cxx`,无文字赘述;每小节内容以表格形式给出,详细描述变更内容,表格列为 项、旧版(A)、新版(B)、是否变更(状态符号 🔄 / 🌓 / ➖,居中对齐)、变更解释(一句话描述)
  - 第四章 兼容性与迁移 : 以表格形式给出,表格列为 项(持久化可读、驱动、控制台)、A → B(是否也需要同时变更,居中对齐)、说明
- 精简 : 能用表则表 ; 不与 Oracle/PG/MySQL 竞品对比
- **工作流程** :
  1. **定位仓库** : 在含 `README.md`、`docs/` 的**本项目根目录**工作 ; Agent 模式须**实际执行**终端 , 不得只列命令
  2. **文档落盘** : 按本章「输出格式」写入 `docs/<主题>/` ( 如 `docs/interval/INTERVAL-05-版本比对.md` ) , 替换 `{输出文件}` 占位 ; 须已读同主题设计/需求及 A/B 版本依据
  3. **本阶段产物** : 仅版本比对 Markdown ; **不**生成 `scripts/` SQL 与 `matrix/` 矩阵 ( 回归脚本引用 **Cxx** 与金样 **GS-xxx** 即可 )
  4. **调用项目** ( 按需 ) : 仅当需验证升级后行为时 , 才 `.\run.ps1` / `.\run-connect.ps1` , 报告在 `reports/`

**约束** :
- 文章中的 `,` , `:` , `/` 等均使用英文并且前后各有一个空格 ; `" "` , `()` , `<>` , `** **`整体而非单个 `"` 或 `(`  以及行内代码块等在前后各加一个空格,如果配合使用则两个符号之间不用空格隔开,例如 : `**()**` 表示包括括号在内的内容加粗,文章中不要出现任何的中文标点符号,例如`小明 **张三** 大明`
- 对于编号列表,要在文档开始以表格的形式总览,在后续章节中详细说明
- 表格中的编号列居中对齐,文档约定的符号例如 ✔️ / 🔄 等也居中对齐
- 当有编号的时候,在文档最前面的章节要有总览,包含编号、说明,之后在后续章节中详细展开描述
- 缺 A/B 设计依据时不编造 **Cxx** 条目
- 不要把三库竞品差异写进版本比对
