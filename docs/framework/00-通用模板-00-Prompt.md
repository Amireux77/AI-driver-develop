# 约束与要求
> "填写 AI 在生成结果的同时受到的【约束】,规范 AI 不会发散性思维而导致生成结果过冗余,【要求】要尽可能的详细"

**示例** : 

**要求** :
- 结构完整 : 列出文档中所必须的所有标准章节
- 逻辑内容 : 要逻辑清晰,各章节之间尽最大限度不要过多的重复,如一定要有联系,用文档名 ( 如《INSERT-02-需求分析》) 或「本小节」引用,勿写「§x.x」( 各类型正文模板章节不一致 )
- 编号列表 : 文章中对每一项需要描述的项均需有编号列表
- 文档信息 : 在文档最开始的位置以引用的格式列出文档的元信息,引用中不要包含表格,基础项包括以下列表,如有需要可补充,基础项用粗体,具体内容不使用粗体
  - 文档版本
  - 编写日期
  - 编写人员
  - 文档定位
  - 适用范围
  - 文档说明
- 内容详尽 : 要尽可能的直观描述 XXX 说明
- 包含内容 : 依据实际文档需求包含以下状态标记与含义
    | 状态标记 | 含义                                                        |
    | :------: | ----------------------------------------------------------- |
    |    ✔️     | 功能已实现,通过规范,可以纳入测试用例中测试正确性            |
    |    ❌     | 功能未实现,待评审决策之后确定该功能是否需要实现             |
    |    ➖     | 库无对应场景 / 保持不变                                     |
    |    ⚠️     | 功能已实现,但行为不符合规范,可能会造成影响,待评审决策后更新 |
    |    🔺     | 部分实现                                                    |
    |    🔄     | 内容变更                                                    |
    |    🌓     | 部分变更                                                    |
- **工作流程** ( 读取任一阶段 Prompt 时均须遵守 ) :
  1. **定位仓库** : 在含 `run.ps1`、`config/`、`db-compat-tester/`、`tools/new-sql-script.ps1` 的**本项目根目录**工作 ; Agent 模式须**实际执行**终端
  2. **环境预检** : 检查 `config/db.properties`、`lib/jdbc/*.jar`、`db-compat-tester/target/db-compat-tester-1.0.0-jar-with-dependencies.jar`
  3. **文档落盘** : 各阶段 Markdown 写入 `docs/<主题>/` , 文件名与「输出格式」一致 ( 如 `INSERT-01`～`INSERT-04`、`INTERVAL-01`～`INTERVAL-05` )
  4. **SQL 落盘** : **用例设计之后** — 按《00-通用模板-00-Prompt-SQL生成》; **§16.1 已实现** → `scripts/<主题>_TYPE/implemented/` ( [docs.xugudb.com](https://docs.xugudb.com/) 语法 ) , **GR/待实现** → `…/gap/` ; 脚手架 `-Scope implemented|gap`
  5. **四库验证** : `.\run-connect.ps1` ; 矩阵 `.\run.ps1 matrix/<文件>.json` → `reports/` ; 创表须删表
  6. **阶段分工** : 可行性/需求/方案/版本比对 → 主要产出文档 ; 用例设计 → 文档 + SQL ( INTERVAL 可 `-VariantCount 13` , INSERT/UPDATE 单场景 )


**约束** : 
- 文章中的 `,` , `:` , `/` 等均使用英文并且前后各有一个空格 ; `" "` , `()` , `<>` , `** **` 以及行内代码块等在前后各加一个空格,如果配合使用则两个符号之间不用空格隔开,例如 : `**()**` 表示包括括号在内的内容加粗,文章中不要出现任何的中文标点符号
- 对于编号列表,要在文档开始以表格的形式总览,在后续章节中详细说明
- 表格中的编号列居中对齐,文档约定的符号例如 ✔️ 等也居中对齐
- 当有编号的时候,在文档最前面的章节要有总览,包含编号、说明,之后在后续章节中详细展开描述
- 编号命名须符合《00-通用模板-00-统一编号体系》, 规则如下 :
  - **主题不写进 FR/TD/AC 等编号** : 测试主题已由输出文件名表达 ( 如 `INSERT-02-需求分析.md` ) , **FR-001**、**TD-01** 等**禁止**再嵌入 `INS`、`INTERVAL` 等主题段 ; 可执行用例使用 **XG-TYPE-&lt;主题&gt;-&lt;域代码&gt;-&lt;序号&gt;** ( 与 INTERVAL 脚本一致 )
  - **允许的前缀** : `FR` 功能需求 , `GR` 缺口需求 , `PD` 待实现 , `AC` 验收 , `TD` 测试域 , `XG` 可执行用例 ( `XG-TYPE-<主题>-<域代码>-<序号>` ) , `GS` 金样 , `G` 仅可行性缺口 , `A`/`Q`/`L`/`C`/`T` 等见统一编号体系
  - **格式示例** : `FR-001` , `GR-015` , `TD-08` , `AC-03` , `XG-TYPE-INSERT-VAL-001` ( 主题缩写在 `TYPE-` 后一段 , 域代码在编号中 )
  - **跨文档引用** : 须带文档名 , 如《INSERT-02》**FR-001** , 避免与其它系列的 **FR-001** 混淆
